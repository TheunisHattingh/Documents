Linux Logrotation Documentation

This documentation is to describe my findings on how to use the logrotate command on linux systems.

Overview

To automate this process you will need the following:

  • log file

  • Configuraation file

  • A cronjob to run the command

Log File

To start with you will need a log file which you want to rotate the contents of. This method of rotating your log files will only work if your logs are only written to a single log file, i.e. one log file that is continuously written to. It can still rotate multiple logs, but will run into hickups when a new log file is created each day. The reason for this is that the rotations will compare the base file that it is pointing at to the ones that have already been rotated so that it can accordingly clean up after itself.

Coniguration File

The configuration file is where most of the work is done as it should contain the location of the log files and the criteria you want to rotate it with. The first thing in your config file should be the path(s) to your log file(s), followed by oyur criteria in the form of logrotate tags.

Paths

The path to your log file can be written in a few ways which I will give an example of.

You can give the direct path to that log file: /home/user/logs/example.log

You can also point to the log by using certain regex expressions:

/home/user/logs/*.log this will also point to any other files in the directory ending with the same extention, e.g. this will work the same as:

/home/user/logs/example1.log
/home/user/logs/example2.log

You can also list multiple files for rotation:

/home/user/logs/example.log
/home/user/logs/example2.log
/home/user2/logs/*.log
/home/user3/logs/*-???.log

Configuration tags

There are a lot of useful tags that canbe used in your config file that can be found with man logrotate, but I will only explain a few that I have used from the following example.

/opt/wso2ei/default/repo/logs/*.log
{
	su wso2 wso2
	daily
	missingok
	rotate 999999
	dateext
	dateformat -%Y-%m-%d-%H
	maxsize 5G
	maxage 7
	compress
	delaycompress
	copytruncate
}

The su wso2 wso2 tag is setting the ownership and the group su {user} {group} ,which the logs should be rotated as and belong to. This might not be needed depending on the permissions of the log files.

The daily tag indicates that the file should rotate on a daily basis and will only rotate if the date has changed since the last rotation. There are also similar tags such as hourly, weekly, monthly, etc.

The missingok tag is used when there is a possibility that there won’t always be log files in a certain directory, otherwise it is not needed.

The rotate tag indicates how many files may be rotated, in this case I used an unreasonably big number just to make sure all the logs get rotated. An example of how this works is that if you used rotate 3 and have 4 log files, only the first 3 log files would get rotated.

The dateext tag indicates that the rotated files that are created must have a date extension to indicate when they were rotated. This is usually followed by the dateformat tag.

The dateformat tag follows the dateext tag and indicates how the date extension should be displayed using tags which can be found on https://linux.die.net/man/3/strftime. From the -%Y-%m-%d-%H used it will display as e.g. -2019-09-20-09 and the dashes (-) are only seperator’s which are used for readability and are not necessary. Depending on your logrotate version you may be able to go into more accurate date extensions including minutes and seconds if needed, but for the version I was working on this was not possible, It would also not be a good idea to set the cronjob to logrotate more often than the last part of your extension as an extra rotation with the same extension will replace the previous logfile.

The maxsize 5G tag indicates that if the log file exceeds 5Gb in size before the next day it should be rotated again. This will only work if the logrotate command is run more than indicated by the daily tag. For instance the config file above has the daily tag but the cronjob is setup to run the logrotate command with this config file every hour, and will only rotate multiple times a day if the log file has exceeded 5Gb in size. This is to avoid clutter and becomes more useful with the compress tag.

The maxage tag is used to indicate that the rotated log files should be removed once they are older that the specified number of days. In the example above maxage 7 is used and once a rotated file is 8 days old it will be removed upon the next rotation.

The compress tag indicates that the rotated log files should be compressed after rotation (uses gzip). This is usally always a good idea to have in your configuration as it can save a lot of drive space depending on the size of your logs.

The delaycompress tag is mostly used for accessability, as it will delay the compression of your rotated log files untill the next rotation. This is useful when you can immediately react to an issue and have immediate access to the most recent log file, otherwise you would need to use the gunzip command to first unzip it and then access the log file.

The copytruncate tag is very important as this makes it possible to rotate the file while it is being written to. It takes everything up to the point where the log is being written to and copies it to the rotated file and removes it from the original.

Cronjob

Firstly you will need to understand o to manually run the logrotate command and it shouldn’t be too hard to understand, as you usually just do the following:

sudo logrotate -v /path/to/config.conf

Using sudo might only be necessary depending on permissions on the files but it usually isn’t a bad idea to use it. The -v or --verbose will give you output of what the rotation has done, and in case there were some errors it will display what went wrong, and hopefully give an indication of what needs to be changed. Sometimes if there is something wrong and you can’t get the file to be rotated it might be necessary to add a -f or --force to force a rotation just to reduce clutter but this shouldn’t be needed once your rotations work corrcetly, I only used this while setting up the rotations and I didn’t have a full understanding of what went wrong and why the rotations didn’t occur correctly.

The basics of setting the time of a cronjob can be tested on https://crontab.guru/ and will usually be in the following format:

#m h d M DoW ( minute, hour, day, month, day of week )
0 */1 * * * {user} {command}

Using the root as the user for running the cronjob is recommended as the ownership and group can be assigned in the config file and this will most of the time avoid any permission conflicts. Also running the logrotate command in the cronjob won’t be exactly the same as you would manually do it as you wil need to point exactly to the logrotate command’s path:

0 */1 * * * root /usr/sbin/logrotate /path/to/config.conf

But for starters it might be better to create a temporary file in your home directory and pipe the output and errors to that file at the beginning until you are sure the cronjob works:

cd /home/user/
touch temp.log

Then in your cronjob:

0 */1 * * * root /usr/sbin/logrotate -v /path/to/config.conf &>> /home/user/temp.log

In this case you might need to change the ownerhip of your temporary log file:

sudo chown root:root /home/user/temp.log

This will not always be necessary but if there are permission conflicts try to set it to the user and group which is in the config file (shouldn’t be an issue in your home directory, but if it is it will be displayed by the verbose output). Whether the logrotation was succesful or not the output should be displayed inside the temp.log file and you can make adjustments accordingly.

Process

The following will be an example of the full process.

  • Let’s say that the log file’s are as follows:

/var/share/backups/backup.log
/var/share/example/audit.log
/var/share/example/gc.log

  • Let’s also make sure that all the log files have the following permissions -rw-rw-r-- which means that the user - group - other have read/write and read permissions respectively, if this is not the case you can change it with sudo chmod 664 /path/to/logfile.log, and you can also read up more on the permissions available at https://www.computerhope.com/unix/uchmod.htm Using ls -larth /path/to/dir the logs should look something like this

  • In the above image the wso2 share indicates the user and group that the file belongs to. To change this for the current example we will be using wso2 as our user and share as our group. To change the logs to belong to this user and group run the following commands:

sudo chown wso2:share /var/share/backups/backup.log
sudo chown wso2:share /var/share/example/audit.log
sudo chown wso2:share /var/share/example/gc.log

  • The log files are now as they should be, normally you won’t need to create them yourself but I wanted to include changing their ownership and permissions in this example. Now we need to create a config file which will rotate these files. Let’s say we want them to rotate daily or when their size exceeds 100Mb, and there should only be 12 days of logs at a time which all have a date extension using the format -YYYY_mm-dd_HH, they should also use delayed compression.

sudo touch /var/share/config.conf
sudo chown root:root /var/share/config.conf
sudo nano /var/share/config.conf

Now you should edit your config file as follows:

/var/share/backups/backup.log
/var/share/example/*.log
{
    su wso2 share
    daily
    rotate 9999
    dateext
    dateformat -%Y_%m-%d_%H
    maxage 12
    maxsize 100M
    compress
    delaycompress
    copytruncate
}

  • The last step is to setup the cronjob

sudo touch /etc/cron.d/examplerotate
sudo chown root:root /etc/cron.d/examplerotate
sudo nano /etc/cron.d/examplerotate

Now you will need to make the cronjob run the logrotate command more than once per day to be able to trigger an extra rotation if the size of the log files exceed their limits, and since the date extension only indicates up to the Hour you can make it run at the start of every hour if you make it check any more frequently there is a chance that some files may be overwritten or lost.

0 */1 * * * root /usr/sbin/logrotate /var/share/config.conf

  • That should do it. THE END.

Extra notes

  • Forgetting can cause issues. Always remember calling commands in a cronjob requires you to point to the command’s bas directory i.e. /usr/sbin/logrotate

  • (This did happen when I started setting up the logs and didn’t get it right the first time) I have noticed that if there is a day that is missed in the rotations and your log files look something like this:

backup.log
backup.log-2019-09-12.gz
backup.log-2019-09-14.gz
backup.log-2019-09-15.gz
backup.log-2019-09-16.gz
backup.log-2019-09-17.gz
backup.log-2019-09-18.gz
backup.log-2019-09-19.gz
backup.log-2019-09-20.gz
backup.log-2019-09-21

Notice that there is no rotation for the 13th.In this case I had the config setup so that the logs can have a maxage of 7 meaning that the rotation for the 12th(there could be more previous files) should have been removed already but it didn’t happen. This is because at this point the 14th rotation is exactly 7 days old and should not yet be removed, it then checks if the next rotation(13th) is there to remove (it’s not) and can’t find it, thus it thinks there is nothing to remove and finishes. Then the following day:

backup.log
backup.log-2019-09-15.gz
backup.log-2019-09-16.gz
backup.log-2019-09-17.gz
backup.log-2019-09-18.gz
backup.log-2019-09-19.gz
backup.log-2019-09-20.gz
backup.log-2019-09-21.gz
backup.log-2019-09-22

Rotation ran and once it noticed the 14th’s rotation is 8 days old it removed it and any older rotations. This means even if there is a day where the server is down and can’t log the rotations will get themselves back in order if they mess up in the same way.
